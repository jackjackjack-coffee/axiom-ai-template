How we built sub-150ms inference
at 10M requests per day

A deep dive into the architectural decisions, caching strategies, and hardware optimizations that let us deliver consistently fast AI inference at massive scale — without breaking the bank.

James Park Staff Infrastructure Engineer

On this page

Background Finding bottlenecks KV cache strategy Dynamic batching Hardware selection Results What’s next

Background

When we launched Axiom in 2024, our p50 inference latency sat around 380ms. For demos, that was fine. For production workloads — real-time customer chat, document processing pipelines, agent loops that chain dozens of calls — it wasn't. Our users were polite about it, but the churn data told the real story.

We set an aggressive internal target: sub-150ms p50, sub-400ms p99, at 10M requests per day without scaling costs proportionally. This post is the story of how we got there.

Finding the bottlenecks

Our first step was brutal honesty about where time was actually going. We instrumented every layer — network ingress, tokenization, queue time, model forward pass, detokenization, response serialization — and built a flame graph across 10,000 sampled requests.

The results were humbling. Model forward pass was only 40% of wall-clock time. The other 60% was split between:

  • Queue wait time (23%) — requests sitting idle waiting for a GPU slot
  • KV cache misses (18%) — recomputing attention for repeated prefixes
  • Network overhead (12%) — request routing through too many hops
  • Tokenization (7%) — surprising, but CPU-bound tokenization was a real bottleneck at scale

This told us exactly where to focus. Optimizing the model itself would yield at best a 40% improvement. Fixing the infrastructure would give us 2× or more.

⚡
Key insight

Model forward pass is often not the bottleneck in production AI systems. Profile your full request path before optimizing the model.

KV cache strategy

The single highest-leverage optimization was prefix caching. Many production workloads share long system prompts — a customer service agent might have 2,000 tokens of context that's identical across every request. Recomputing attention over those 2,000 tokens on every call is pure waste.

We built a distributed KV cache using a content-addressed key scheme: SHA-256 of the normalized token sequence maps to a GPU-memory-resident attention cache entry. Eviction is LRU with a frequency-weighted scoring function that keeps hot prefixes resident even under memory pressure.

class PrefixCache: def get_or_compute(self, tokens: list[int]) -> KVState: key = sha256(normalize(tokens)) if cached := self.store.get(key): self.stats.record_hit(len(tokens)) return cached state = self.model.compute_kv(tokens) self.store.set(key, state, ttl=3600) return state

Cache hit rate reached 73% within the first week on production traffic. For requests with long shared prefixes, this alone cut latency by 45%.

Dynamic batching

Static batching — wait for N requests, then process together — is simple but brutal for tail latency. If you're waiting for a batch of 8 and only 3 requests arrive in 50ms, either you wait (hurting latency) or you process early (wasting GPU utilization).

We implemented a dynamic batching controller that adjusts batch size and wait time based on current queue depth and per-request priority. The algorithm is a modified version of Orca's iteration-level scheduling, extended with priority lanes for Pro and Enterprise customers.

This reduced p99 latency by 38% while actually increasing GPU utilization from 61% to 84%.

Hardware selection

We benchmarked H100 SXM, H100 PCIe, A100 80GB, and L40S across our workload mix. The answer wasn't obvious — H100 SXM wins on raw throughput, but our workload mix includes many short requests where NVLink bandwidth doesn't help and the price premium doesn't justify itself.

We landed on a heterogeneous fleet: H100 SXM clusters for long-context and fine-tuned model inference, L40S clusters for standard inference where the price-per-token is 40% better. Traffic is routed to the appropriate cluster based on request characteristics detected at ingress.

What’s next

We're not done. The next phase focuses on speculative decoding — using a small draft model to predict likely token sequences, then verifying in parallel with the full model. Early experiments suggest another 40–60% latency reduction for standard-length outputs.

We're also exploring disaggregated prefill/decode — splitting the compute-heavy prefill phase from the memory-bandwidth-bound decode phase onto different hardware. This is bleeding edge territory, but the potential gains are significant.
